**Версия:** 1.0

**Автор:** Senior Front-end Developer  

---

## 1. Введение

### 1.1. Назначение проекта
«Team Pulse» — это корпоративный веб-дашборд для мониторинга активности команды в реальном времени. Система позволяет руководителям и участникам проектов отслеживать текущий статус задач, загрузку сотрудников, получать уведомления о критических событиях и оперативно управлять рабочим процессом.

### 1.2. Цели и задачи
- Предоставить единую точку входа для визуализации ключевых метрик команды.
- Автоматизировать процесс оповещения о просроченных задачах и изменениях статусов.
- Обеспечить высокую производительность и отзывчивость интерфейса даже при большом количестве одновременных пользователей.
- Продемонстрировать современные подходы к разработке на Vue 3 / Nuxt 4 (гибридный рендеринг, серверные API, WebSocket, микросервисная архитектура на Nitro).

### 1.3. Целевая аудитория
- **Менеджеры проектов** – для контроля загрузки и сроков.
- **Рядовые сотрудники** – для ведения личных задач и просмотра активности коллег.
- **Администраторы** – для управления пользователями и ролями.

## 2. Функциональные требования

### 2.1. Модуль аутентификации и авторизации
- Регистрация новых пользователей (email + пароль) с подтверждением по email.
- Вход в систему (JWT-токены, refresh-токены).
- Восстановление пароля.
- Разграничение ролей: **admin**, **manager**, **user**.
- Защита всех маршрутов, кроме `/auth/*`, с помощью глобального middleware.

### 2.2. Личный кабинет и профиль
- Просмотр и редактирование профиля (аватар, имя, должность, контакты).
- Смена пароля.
- Отображение списка задач, назначенных пользователю.

### 2.3. Дашборд (главная страница)
- Виджет «Активность команды» – лента последних действий (создание задачи, комментарий, изменение статуса) с фильтром по проектам.
- Виджет «Загрузка участников» – диаграмма (столбчатая или круговая) количества задач на каждого сотрудника.
- Виджет «Просроченные задачи» – список задач с истекшим дедлайном, подсвеченных красным.
- Виджет «Статистика по проектам» – график выполнения задач за неделю/месяц (линейный график).
- Виджет «Мои задачи» – краткий список задач текущего пользователя с приоритетами.

### 2.4. Управление задачами (CRUD)
- Создание задачи (название, описание, исполнитель, дедлайн, приоритет, проект).
- Редактирование задачи (изменение любого поля).
- Удаление задачи (мягкое удаление с возможностью восстановления администратором).
- Изменение статуса задачи: `новая`, `в работе`, `на проверке`, `завершена`, `отменена`.
- Комментирование задач (в реальном времени с оповещением участников).
- Прикрепление файлов к задаче (до 10 МБ, формат: изображения, PDF, DOCX).

### 2.5. Поиск и фильтрация
- Глобальный поиск по задачам, пользователям, проектам с автодополнением.
- Фильтрация задач по статусу, исполнителю, проекту, дате создания/дедлайну.
- Сохранение пользовательских фильтров в локальном хранилище.

### 2.6. Уведомления
- Система уведомлений в реальном времени (WebSocket) о новых задачах, комментариях, изменении статуса.
- История уведомлений с пометкой о прочтении.
- Возможность подписаться на уведомления по конкретному проекту.

### 2.7. Административная панель (доступна только admin)
- Управление пользователями (блокировка/разблокировка, смена роли).
- Просмотр логов активности (кто, когда, что делал).
- Настройка общих параметров системы (дефолтные приоритеты, статусы).

## 3. Технические требования

### 3.1. Стек технологий
| Компонент | Технология | Версия |
|-----------|------------|--------|
| Фреймворк | Nuxt | 4.x |
| Язык | TypeScript | 5.x |
| UI-библиотека | Naive UI (или PrimeVue) | последняя |
| Управление состоянием | Pinia | 2.x |
| Серверный движок | Nitro | 3.x (с поддержкой crossws) |
| База данных | SQLite (dev) / Turso (prod) | – |
| ORM | Drizzle | 0.38+ |
| Валидация | Zod + VeeValidate | 4.x / 4.x |
| Графики | Chart.js (vue-chart-3) | 4.x |
| WebSocket | crossws (встроенный в Nitro) | – |
| Тестирование | Vitest + @vue/test-utils | 2.x |
| Сборка | Vite (с Rolldown – опционально) | 5.x |
| Деплой | Vercel (preset: vercel-edge) | – |

### 3.2. Архитектурные принципы
- **Модульность** – каждый функциональный блок (задачи, пользователи, уведомления) находится в отдельной директории и экспортирует свои композаблы, компоненты и стор.
- **Композаблы как основной способ логики** – вся бизнес-логика вынесена в `composables/` (useTasks, useAuth, useWebSocket и т.д.).
- **Серверные API** – все эндпоинты реализованы через Nitro с использованием `event.h3` и автоматической типизацией для `$fetch`.
- **Гибридный рендеринг** – для каждой страницы задаётся своя стратегия:  
  - `/dashboard` – SSG с ISR (перегенерация каждые 60 секунд).  
  - `/tasks/**` – SSR (динамические данные).  
  - `/auth/**` – SPA (без серверного рендеринга, для скорости).
- **Кеширование** – использование `useFetch` с ключами и стратегией SWR.
- **Реальное время** – через WebSocket (Nitro crossws) для ленты и уведомлений.

### 3.3. Безопасность
- JWT-токены хранятся в `httpOnly` куках (безопасные, защищённые от XSS).
- Все серверные роуты защищены middleware, проверяющим роль пользователя.
- Валидация входных данных на сервере с помощью Zod.
- Санитизация HTML-контента в комментариях.
- Ограничение частоты запросов (rate limiting) для `/auth/login` и `/auth/register`.

### 3.4. Производительность
- Целевой показатель First Contentful Paint (FCP) < 1.2 секунды для SSR-страниц.
- Время до интерактивности (TTI) < 2.5 секунд.
- Оптимизация изображений через `@nuxt/image`.
- Автоматическое разделение кода (code splitting) на уровне страниц и компонентов.
- Использование `defineAsyncComponent` для тяжелых виджетов (графики).

## 4. Пользовательские сценарии (User Stories)

1. **Как менеджер**, я хочу видеть на дашборде общую загрузку команды, чтобы распределять задачи равномерно.
2. **Как разработчик**, я хочу получать уведомления о новых задачах и комментариях в реальном времени, чтобы оперативно реагировать.
3. **Как администратор**, я хочу блокировать пользователей и просматривать историю действий для аудита.
4. **Как пользователь**, я хочу фильтровать задачи по проекту и статусу, чтобы быстро находить нужное.
5. **Как менеджер**, я хочу создавать задачи с дедлайном и назначать исполнителей, чтобы контролировать сроки.
